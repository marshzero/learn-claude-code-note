Chapter 01

让LLM调用工具

Agent Loop的设计

1. 问题引入

当我们让大语言模型（LLM）帮我们完成一个任务时，比如"读取当前目录下的所有文件"，会发生什么？

LLM能做什么

分析问题、理解意图、给出步骤指引

LLM不能做什么

无法真正执行任何操作——不知道当前目录有什么文件

核心问题：如何让LLM真正"动手"做事，而不是只给建议？

2. 解决方案

答案出乎意料地简单——用循环

User
prompt
→
LLM
→
Tool
execute
↓ tool_result ↑

(loop until stop)

循环逻辑

  • 用户提出请求
  • LLM分析请求，决定是否需要调用工具
  • 如果需要，执行工具并获得结果
  • 把结果反馈给LLM，让它继续分析
  • 重复直到LLM认为任务完成

2.1 核心设计

整个循环依赖两个关键设计：

stop_reason

何时停止

tool_use 块

传递什么任务

stop_reason —— 何时停止

LLM的回复中包含一个关键字段 stop_reason，它告诉程序LLM为什么停止生成：

  • tool_use — LLM想要调用工具，循环继续
  • end_turn — LLM认为任务完成，循环终止
  • max_tokens — 达到输出上限，可能是异常终止
if response.stop_reason != "tool_use": return # LLM说任务完成了，退出循环

tool_use 块 —— 传递什么任务

LLM的回复是一个内容块（block）列表，每个块可能是：

  • text — 纯文本回复
  • tool_use — 工具调用请求
for block in response.content: if block.type == "tool_use": tool_name = block.name tool_input = block.input tool_id = block.id

2.2 示例数据流转

用户请求
{"role": "user", "content": "List all Python files..."}
LLM响应
{"stop_reason": "tool_use", "content": [{"type": "tool_use", "name": "bash", ...}]}
工具执行结果
{"role": "user", "content": [{"type": "tool_result", "tool_use_id": "tool_123", ...}]}
最终LLM响应
{"stop_reason": "end_turn", "content": [{"type": "text", "text": "完成!"}]}

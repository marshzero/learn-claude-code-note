Chapter 01

Agent Loop

让LLM真正"动手"做事

LLM的困境

🤔
LLM 能做什么

分析问题、理解意图、给出步骤指引

❌
LLM 不能做什么

无法真正执行操作——不知道当前目录有什么文件

传统方案：LLM给出步骤指引，用户手动执行，再把结果粘贴回给LLM。这中间的"翻译"工作全部由人完成，LLM只是一个"顾问"。

核心问题：如何让LLM真正"动手"做事，而不是只给建议？

解决方案：用循环

循环的逻辑

1
2

  • 用户提出请求
  • LLM分析请求，决定是否需要调用工具
  • 如果需要，执行工具并获得结果
  • 把结果反馈给LLM，让它继续分析
  • 重复直到LLM认为任务完成

核心设计：两大支柱

🛑
stop_reason

何时停止

📦
tool_use 块

传递什么任务

整个循环依赖这两个关键设计。用一个条件判断控制整个流程的退出。

stop_reason — 何时停止

LLM的回复中包含一个关键字段，告诉程序LLM为什么停止生成：

tool_use

LLM想要调用工具
循环继续

end_turn

LLM认为任务完成
循环终止

max_tokens

达到输出上限
可能异常

if response.stop_reason != "tool_use": return # LLM说任务完成了，退出循环

tool_use 块 — 传递什么任务

LLM的回复是一个内容块列表，每个块可能是：

📝
text

纯文本回复

🔧
tool_use

工具调用请求

for block in response.content: if block.type == "tool_use": tool_name = block.name tool_input = block.input tool_id = block.id

数据示例：用户请求

{ "role": "user", "content": "List all Python files in this directory" }

用户用自然语言提出请求，这个请求会被加入到messages数组中发送给LLM。

数据示例：LLM响应 (tool_use)

{ "stop_reason": "tool_use", "content": [ { "type": "tool_use", "name": "bash", "id": "tool_123", "input": { "command": "ls -la *.py" } } ] }

LLM分析后决定需要调用工具，stop_reason="tool_use"表示循环继续。

数据示例：工具执行结果

{ "role": "user", "content": [ { "type": "tool_result", "tool_use_id": "tool_123", "content": "hello.py\ngreet.py\nmain.py" } ] }

工具执行结果被封装成tool_result，通过tool_use_id关联到对应的调用请求，然后反馈给LLM。

数据示例：最终响应

{ "stop_reason": "end_turn", "content": [ { "type": "text", "text": "当前目录下有3个Python文件： hello.py、greet.py、main.py" } ] }

stop_reason="end_turn" 表示LLM认为任务完成，循环终止！

实战尝试

python agents/s01_agent_loop.py

试试这些命令：

  • Create a file called hello.py that prints "Hello, World!"
  • List all Python files in this directory
  • What is the current git branch?

你会看到LLM如何一步步思考、调用工具、获取结果、再继续思考的完整过程。

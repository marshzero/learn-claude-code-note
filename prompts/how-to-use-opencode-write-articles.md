# 如何使用 Opencode 写文章

## 概述

本文分享使用 Opencode AI 编程助手撰写技术文档的完整流程和经验。通过一个实际的 12 章节 Agent 教程的写作过程，展示如何高效地用 Opencode 完成文章创作。

- Agent: OpenCode

- Model: Minimax-2.7

- 时长: 1分钟生成1篇文章

从生成第一篇文章，整理提示词，生成其他文章，中间对文章进行审视，总耗时2小时。

## 项目背景

我需要为一个 learn-claude-code 教程生成 12 章节的文章。每章对应一个 Python 示例代码，文章需要包含：
- 问题引入（为什么需要这个功能）
- 解决方案（核心设计）
- 示例数据（详细的数据流转）
- 实战尝试（代码运行和观察点）

## 写作流程

### 1. 理解需求：先读参考文档

每次写新章节前，先阅读原始的英文文档和代码：

```bash
# 读取英文参考文档
read docs/en/s02-tool-use.md

# 读取对应的 Python 代码
read agents/s02_tool_use.py
```

**关键点**：

- 先理解代码逻辑，不要只看文档
- 特别注意代码中的数据结构、流程控制
- 注意文档和代码可能不完全同步

### 2. 确定结构：遵循模板

每章文章采用统一结构：

```
## 第X章：标题

### 1. 问题引入
- 具体场景
- 核心矛盾

### 2. 解决方案
#### 2.1 核心设计
- 架构图或流程图
- 关键代码片段

#### 2.2 示例数据
- 工具Schema
- 完整交互示例
- 数据流转详解

### 3. 实战尝试
- 运行命令
- 尝试任务
- 观察点和思考点
```

### 3. 生成初稿

根据代码生成文章：

```bash
write content="## 第二章：Tool Use...\n..."
filePath="chapter02-tool-use.md"
```

**经验**：
- 先写一个完整章节，理解风格
- 后续章节参考前面的风格
- 不要一次性写太长，容易出错

### 4. 迭代修改：用户反馈驱动

Opencode 的优势是可以边写边改：

```
用户: "火后即忘这个词不够直观"
助手: 修改措辞

用户: "示例数据不对，Turn 4的流程有问题"
助手: 重新理解代码逻辑，修正示例

用户: "缺少reject场景"
助手: 补充完整场景
```

**关键对话模式**：

```bash
# 指出问题
read chapter06-context-compact.md
# 看完后说：Turn 4的流程有问题，micro_compact在LLM调用之前执行

# 修改
edit filePath="chapter06-context-compact.md"
oldString="原内容"
newString="修正后的内容"
```

## 核心技巧

### 1. 复杂示例要画图

数据流转用 ASCII 图表示：

```
Turn 1:
  User: "Read a.txt"
  messages: [...]
  micro_compact检查: 0个tool_result，不压缩
  LLM决定调用 read_file(a.txt)
  → 执行tool，生成tool_result: "content of a.txt"
  messages增加: [..., t1]
```

### 2. 渐进式展示

不要一次给所有信息：

```bash
# 先写简单版本
"""
初始状态:
[ ] #1: Design API
[ ] #2: Frontend dev (blocked by: [1])
[ ] #3: Backend dev (blocked by: [1])

Phase 1: task_1完成
[x] #1: Design API
[ ] #2: Frontend dev  ← 可执行
[ ] #3: Backend dev   ← 可执行
"""
```

### 3. 追问细节

用户没提到的也要考虑周全：

```bash
用户: "示例数据不够复杂"
助手: "好的，我重新设计一个7任务的DAG，展示并行和汇聚依赖"
```

### 4. 修正后验证

修改完要确认：

```bash
# 查看修改后的内容
read chapter07-task-system.md
# 确认没有重复内容、逻辑正确
```

## 常见问题处理

### 1. 示例太复杂

```bash
# 用户反馈
"例子又太复杂了，而且你画的图也不对"

# 响应
"明白，精简为4个任务，重点体现并行和汇聚依赖"

# 修改策略
- 减少任务数量
- 画正确的依赖图
- 保留核心概念
```

### 2. 概念难以理解

```bash
# 用户反馈
"没看懂shutdown问题是什么"

# 响应
"让我更直白地解释shutdown问题"

# 改写策略
- 用具体场景代替抽象描述
- 加"一个具体场景"段落
- 用表格对比前后差异
```

### 3. 核心设计看不懂

```bash
# 用户反馈
"又看不懂核心设计了"

# 响应
"让我简化核心设计部分"

# 改写策略
- 减少图和术语
- 用实际消息对话展示
- 一句话总结本质
```

## 工作流程总结

```
1. 读取参考文档和代码
   ↓
2. 理解核心逻辑
   ↓
3. 生成初稿
   ↓
4. 用户阅读并反馈
   ↓
5. 根据反馈修改
   ↓
6. 重复 4-5 直到满意
   ↓
7. 继续写下一章
```

## 经验清单

| 阶段 | 技巧 |
|------|------|
| 阅读代码 | 先理解逻辑，不要只看文档 |
| 生成初稿 | 参考已有章节风格，保持一致 |
| 示例数据 | 用实际数字，少用泛指 |
| 迭代修改 | 每次改一个问题 |
| 确认完成 | 读一遍确保逻辑正确 |

## 总结

使用 Opencode 写文章的关键：
1. **参考实际代码**，不要只依赖文档
2. **渐进式生成**，不要一次写太长
3. **重视用户反馈**，边写边改
4. **保持结构一致**，方便阅读

Opencode 特别适合技术文档写作，因为可以随时运行代码验证、修改后立即确认。比起传统的"写完再改"模式，这种对话式写作更加高效和精确。
